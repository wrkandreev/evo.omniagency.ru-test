# evocms-vibecode-starter

База знаний и рабочий starter для вайбкодинга на проектах с `Evolution CMS CE`.

## Что Это За Репозиторий

Этот репозиторий собирает практические правила, документацию, примеры и повторно используемые workflow для работы с реальными Evo-проектами вместе с AI-агентами.

Это не дистрибутив Evo и не стартовая тема сайта.
Это репозиторий контекста:

- как безопасно разбирать Evo-проекты
- как восстанавливать шаблонный контекст
- как работать с TV, `MultiTV`, `ClientSettings`, `templatesedit` и selector-полями
- как организовывать shared-логику вокруг `BaseController` и `Helper.php`
- как разворачивать свежий Evo `3.x` проект под этот workflow

## Зачем Он Нужен

В `Evolution CMS CE` важная часть структуры проекта хранится в базе данных.
Из-за этого файлового кода недостаточно, чтобы понять проект полностью.

Для AI-assisted работы это создает особую проблему: агенту нужны не только исходники, но и надежная карта того, как реально устроены шаблоны, TVs, контроллеры, manager-конфигурация и дополнительные проектные слои.

Этот репозиторий - попытка собрать такую карту в переиспользуемом виде.

## Главная Идея

Для новых проектов предпочтительный стандарт восстановления шаблонного контекста:

- `https://github.com/wrkandreev/evocms-template-registry`

Если registry установлен на проекте, его нужно использовать как основной source of truth для шаблонного контекста.
Если его нет, этот репозиторий помогает задокументировать и проверить проект вручную.

## Что Внутри

- `AGENTS.md` - входная точка для агентов и набор верхнеуровневых правил
- `core/custom/docs/` - system of record для проектных знаний
- `core/custom/examples/` - эталонные reference-примеры
- `core/custom/skills/` - черновые playbook'и для повторяющихся задач
- `core/custom/decisions/` - короткие ADR с ключевыми решениями
- `core/custom/plans/` - operational memory для больших задач и исследований

## Рекомендуемый Порядок Чтения

1. `AGENTS.md`
2. `core/custom/docs/README.md`
3. `core/custom/docs/golden-rules.md`
4. `core/custom/docs/fresh-install.md`
5. `core/custom/docs/template-context.md`

## Ключевые Принципы

- не доверять только файловой структуре на Evo-проектах
- проверять активную реализацию на живом проекте
- не считать `.sample` и `.example` активной конфигурацией по умолчанию
- не считать `*_old`, `backup`, `views_old` или `Main_old` активным кодом по умолчанию
- различать `selector installed` и `selector used`
- хранить проектные секреты в `core/custom/define.php`, а в git коммитить только `core/custom/define.php.example`
- для новых проектов предпочитать Evo `3.x`, пакет `Main` и новый controller approach с `TemplateController`, `process()` и `addViewData()`

## Workflow Для Свежего Проекта

Предпочтительный bootstrap flow для нового проекта:

1. установить свежий Evo `3.x`
2. использовать `https://github.com/evocms-community/installer`
3. использовать CLI режим installer там, где он доступен
4. создать пакет `Main`
5. установить `evocms-template-registry`
6. использовать новый подход к template controllers

Подробности лежат в `core/custom/docs/fresh-install.md`.

## Как Использовать На Реальном Проекте

1. Скопировать в проект `AGENTS.md` и каталог `core/custom/` из этого репозитория.
2. На живом проекте сначала прочитать `AGENTS.md`, потом `core/custom/docs/README.md`.
3. Проверить, установлен ли `evocms-template-registry`.
4. Если registry не установлен, рекомендовать установить его до серьезной работы с шаблонами, TV и ресурсами.
5. Проверить реальные проектные слои: `ClientSettings`, `templatesedit`, `MultiTV`, selector-поля, `ajax/`, интеграции, cron, доступы.
6. Отделить активную реализацию от исторической: не считать `*_old`, `backup`, `.sample`, `.example` источником истины без проверки.
7. Только после этого начинать кодовые изменения.

Для больших задач имеет смысл:

- завести план в `core/custom/plans/active/`
- зафиксировать устойчивые решения в `core/custom/decisions/`
- обновить docs, если на проекте обнаружился новый подтвержденный паттерн

## Текущий Статус

Этот репозиторий собирается на основе наблюдений по реальным проектам и калибруется на живых установках.

Часть паттернов уже подтверждена на нескольких проектах.
Часть остается provisional и должна проверяться отдельно на каждом живом проекте.

Карта уверенности находится в `core/custom/docs/quality.md`.

## Для Кого Этот Репозиторий

- для разработчиков, работающих с `Evolution CMS CE`
- для команд, использующих AI-агентов для разбора и доработки проектов
- для тех, кто хочет держать переиспользуемую базу знаний по проекту, а не повторять один и тот же контекст в каждом чате

## Важно

- browser-triggered deploy здесь считается custom workflow, а не стандартом Evo
- документация оптимизирована под реальную работу с проектами, а не под абстрактные demo-примеры
- репозиторий предполагается развивать дальше по мере разбора новых живых проектов
